# Roadmap: единый мессенджер на базе Sunrise + SSO (cotton-id)

> Статус: черновик плана (v1). Код ещё не пишется — это согласованная дорожная карта.
> Решения, зафиксированные с заказчиком:
> 1. **tinode** — только ребрендинг имени (протокол и SDK сохраняем).
> 2. **Бэкенд** — оставляем зрелый Go-бэкенд Sunrise, добавляем OIDC-интеграцию.
> 3. Сначала этот план, затем реализация по этапам.

---

## 1. Цель и честная трактовка «уровня Telegram»

Полный паритет с Telegram — это годы работы большой команды (секретные чаты, каналы на
миллионы подписчиков, боты-платформа, стикеры/анимации, своя инфраструктура CDN и т.д.).
Поэтому цель формулируем реалистично:

**Собрать единый продукт** — мессенджер Sunrise с входом через корпоративный SSO (cotton-id),
с современным Svelte-клиентом, доведённым до уровня «повседневного мессенджера» (Telegram-like
по ключевым сценариям), а не побитового клона Telegram.

Целевой набор фич клиента (MVP → расширение) определён в §6.

## 2. Текущее состояние (что уже есть)

### 2.1. `chat/` — бэкенд Sunrise (Go), форк Tinode
Зрелый сервер. Уже реализовано на уровне бэкенда:

| Возможность | Где | Статус |
|---|---|---|
| P2P и групповые чаты (topics) | `server/topic.go`, `init_topic.go` | ✅ |
| Presence (онлайн/оффлайн), typing, read-receipts | `server/pres.go` | ✅ |
| Загрузка/раздача файлов и медиа | `server/hdl_files.go` | ✅ |
| Аудио/видео звонки (WebRTC-сигналинг) | `server/calls.go` | ✅ |
| Push-уведомления (FCM) | `server/push.go` | ✅ |
| Rich text (Drafty) | SDK + datamodel | ✅ |
| Кластеризация / горизонтальное масштабирование | `server/cluster*.go` | ✅ |
| Транспорты: WebSocket, long-poll, gRPC | `server/hdl_*.go` | ✅ |
| Pluggable БД: MySQL, PostgreSQL, MongoDB, RethinkDB | `server/db/*` | ✅ |
| Pluggable аутентификация: `basic`, `token`, `code`, `rest`, `anon` | `server/auth/*` | ✅ |

**Вывод:** бэкенд закрывает большую часть «движка мессенджера». Доработки точечные
(см. §4 SSO, §6 серверные пробелы).

### 2.2. `sso/` — cotton-id (OIDC Identity Provider)
Production-ready IdP на Go 1.25 + chi + PostgreSQL, делегирует OAuth2/OIDC-протокол **Ory Hydra**.
Уже есть: email/пароль, passkeys, social login, self-service аккаунта, admin-консоль (RBAC),
audit-лог, Prometheus/Grafana, SMTP. Spec-first (OpenSpec).

**Роль в продукте:** единая точка входа. Пользователь логинится в cotton-id → получает
identity, мессенджер ему доверяет.

### 2.3. `webapp-svelte/` — новый клиент (Svelte 5 + Vite)
Тонкий скелет (~775 строк): обёртка над `tinode-sdk` + 5 экранов
(`Login`, `Register`, `ChatList`, `ChatView`, `Settings`) и горстка компонентов.
Это стартовая площадка для целевого клиента.

### 2.4. `webapp/` — старый клиент (React)
Полнофункциональный, но легаси. **Решение:** не развиваем; используем как референс фич для
переноса в Svelte-клиент, затем выводим из эксплуатации.

## 3. Целевая архитектура

```
            ┌─────────────┐     OIDC (authorization_code + PKCE)     ┌──────────────┐
            │  cotton-id  │◄────────────────────────────────────────►│  Ory Hydra   │
            │   (SSO IdP) │     login/consent challenge               │ (OIDC engine)│
            └──────┬──────┘                                           └──────┬───────┘
                   │ identity (users, sessions)                              │ ID/access token (JWT)
                   ▼                                                         ▼
        ┌──────────────────────┐        auth-схема rest/oidc        ┌───────────────────────┐
        │  webapp-svelte (SPA)  │──── login(token) ────────────────►│  Sunrise backend (Go) │
        │  Svelte 5 + Vite      │◄─── session token, WS/gRPC ───────│  topics, presence,    │
        │  целевой мессенджер   │                                   │  files, calls, push   │
        └──────────────────────┘                                   └───────────┬───────────┘
                                                                                │
                                                                    PostgreSQL / выбранная БД
```

Ключевая идея: **cotton-id владеет идентичностью, Sunrise — мессенджингом.** Мост между ними —
auth-схема на стороне Sunrise (§4).

## 4. Интеграция SSO ↔ мессенджер (центральная задача)

Бэкенд Sunrise аутентифицирует через pluggable-интерфейс `auth.AuthHandler`
(`chat/server/auth/auth.go:232`), схемы регистрируются по имени. Есть два пути подключить cotton-id:

### Вариант A — схема `rest` + тонкий адаптер (быстрый старт)
Sunrise уже умеет схему `rest` (`chat/server/auth/rest/`, пример `chat/rest-auth/auth.py`):
он POST-ит секрет на внешний HTTP-сервис, тот валидирует и возвращает user-record.
Пишем маленький адаптер перед cotton-id/Hydra: он принимает OIDC-токен от клиента,
валидирует его у Hydra (`/userinfo` или интроспекция) и отвечает Sunrise в его формате.

- **Плюсы:** ноль изменений в ядре Sunrise, только конфиг + ~1 небольшой сервис.
- **Минусы:** лишний сетевой хоп, ещё один сервис в деплое.

### Вариант B — нативная auth-схема `oidc` (рекомендуемая цель)
Новый пакет `chat/server/auth/oidc/auth_oidc.go`, реализующий `AuthHandler`:
- `Authenticate(secret)` принимает ID-token (JWT) от Hydra, **локально** валидирует подпись
  через JWKS (`github.com/coreos/go-oidc`), проверяет `iss`/`aud`/`exp`.
- При первом входе — заводит пользователя Sunrise по `sub`/email (по образцу логики `rest`).
- `GenSecret()` делегирует выдачу сессионного токена существующему `token`-хендлеру.
- Конфиг в `sunrise.conf`: `provider_url`, `client_id`, `allowed_audiences`, кеш JWKS.

- **Плюсы:** без лишних сервисов, валидация локальная (быстро), чистая интеграция.
- **Минусы:** ~день-два Go-разработки + тесты.

**Рекомендация:** начать с **A** для быстрого end-to-end сквозного входа, затем заменить на **B**
как постоянное решение. Обе совместимы — клиент в обоих случаях логинится одним токеном.

### Поток входа (целевой)
1. Svelte-клиент уводит пользователя в cotton-id (`/oauth2/auth` Hydra, PKCE).
2. Пользователь логинится в cotton-id, Hydra выдаёт ID-token.
3. Клиент вызывает `tinode.login('oidc', '<id_token>')` (вместо `basic`).
4. Sunrise валидирует токен (через A или B), отдаёт сессионный токен Sunrise.
5. Дальше всё как обычно: WS/gRPC, topics, сообщения.

### Регистрация relying-party
Зарегистрировать мессенджер как OAuth-клиент в cotton-id/Hydra (redirect_uri Svelte-клиента,
scopes `openid profile email`). См. `sso/docs/RUNBOOK.md`.

## 5. Ребрендинг tinode → Sunrise (фиксированный, малый объём)

Цель: убрать имя `tinode` из исходников и UI, протокол/SDK оставить.

Объём (вне собранных `umd/`-бандлов и `node_modules`):
- `chat/server/*` — 7 упоминаний в коде/тестах (`main.go`, `calls.go`, `store/types/uidgen.go`,
  4 теста БД). Замена строк/комментариев, без изменения логики.
- `webapp-svelte/src/*` — 9 упоминаний; ключевое — переименовать `lib/tinode.js` →
  `lib/messaging.js` (или `lib/sunrise.js`) и поправить импорты в 5 view.
- npm-зависимость `tinode-sdk` в `package.json` **оставляем** (это сам протокол); имя пакета —
  внешнее, его не трогаем. Своя обёртка просто не выставляет наружу слово «tinode».
- Собранные бандлы `chat/static/umd/*`, `webapp/umd/*` — артефакты, перегенерируются сборкой;
  правкам не подлежат.

## 8. Риски и открытые вопросы
- **Бренд/имя продукта** — нужно решение заказчика (см. §5).

- **Объём «Telegram-уровня»** — фиксируем MVP по §6, остальное — итеративно.
